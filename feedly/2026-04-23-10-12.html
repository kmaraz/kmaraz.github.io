Using SELECT FOR UPDATE To Synchronize MySQL Row Access In ColdFusion

Ben Nadel uses a small ColdFusion app to turn MySQL locking reads into an application-level synchronization tool, showing where `FOR SHARE` and `FOR UPDATE` belong in the data-access layer and how they act as workflow choke points during multi-step transactions.

  • `FOR SHARE` keeps concurrent readers moving while blocking writers and exclusive lockers on the same rows.
  • `FOR UPDATE` turns a selected row into an exclusive coordination point for later writes and dependent child-row work.
  • Locking reads do not block ordinary non-locking selects, so concurrency costs stay concentrated around mutation paths.
  • Row locking only works as intended when every competing workflow agrees to lock the same record before mutating related state.

Columnar Storage is Normalization

Justin Jaffray reframes columnar storage as an extreme form of normalization: each column behaves like its own narrow table keyed by row position, so reconstructing rows is logically the same operation as a join. The framing is useful because it connects storage layout to familiar relational reasoning instead of treating columnar formats as a separate conceptual world.

  • A columnar representation can be modeled as one implicit key plus one attribute per table, with ordinal position standing in for the key.
  • Reading a full row from columnar storage is conceptually a join across per-column tables.
  • The model explains why column stores favor scans, projections, and analytics that touch few columns across many rows.
  • Thinking in normalization terms makes physical layout choices easier to reason about with ordinary relational abstractions.
